# 案例/经验知识沉淀子系统 — 设计文档（spec）

- 日期：2026-06-09
- 范围：一线人员上传检修案例/经验，经审核后纳入知识图谱（赛题功能需求第 4 条）
- 涉及仓库：weixiu(Java 编排/落库/图谱) · FixAgent(Python LLM/向量/OCR/ASR) · fix-(前端)

## 1. 目标与赛题对应

实现赛题第 4 条：「支持一线人员上传检修案例、经验总结等内容，经审核后纳入知识图谱，实现知识实时更新；同时支持手动标注与修正大模型输出结果，优化系统适配性。」

逐句落地：
- **上传** = 四个输入通道（任务沉淀 / 文件 / 笔记拍照 / 语音口述）。
- **经审核** = 合规 LLM 闸门 + 老师傅确认 + 管理员审核（双人审核）。
- **纳入知识图谱** = 审核通过后写 CaseRecord 节点 + 连边（Device、RECORDED→Fault）。
- **实时更新** = approve 即时向量化 + 入图。
- **手动标注与修正大模型输出** = AI 草稿可被老师傅修改、管理员审核时可再编辑修正。
- **优化适配性/质量** = Basic Reflection 自我修正 + 合规闸门。

## 2. 核心架构：统一主干 + 可插拔输入通道

所有通道最终汇聚到同一条主干，主干只实现一遍：

```
[输入通道] → AI 提取/起草(含 Basic Reflection 自我修正一轮)
          → 合规 LLM 闸门(相关性 + 合规性, 任一不过即拦, 返回理由)
          → 老师傅修改确认(提交) → 待审队列(status=pending)
          → 管理员审核(可编辑修正) → approve
          → 向量化(失败可重试但阻塞通过) + 连边(Device / RECORDED→Fault) 入图谱
```

设计原则：
- **通道是适配器**：每个通道只负责把原始输入（任务/文件/图片/语音）转成「结构化案例草稿」，之后完全复用主干。新增通道不改主干。
- **图谱写入收口在 Java**：沿用项目既有约定（Python 只产 JSON / 算向量，Neo4j 落库与连边在 Java）。
- **审核 gate 用 `status` 字段**：CaseRecord 是独立节点类型（非 manual_auto 自动抽取链路），用 `status` 实现 pending→approved，而非 GraphIngest 的 `verified`。
- **检索是沉淀的"出口"，必须同期建**：沉淀若不能被 RAG 召回 / 用户查看，则毫无意义。经核查，现有检索（`GraphQueryServiceImpl` path/search → graph_java_tool）只向量召回 Fault/Component→遍历 Solution，**完全不召回 CaseRecord**；前端图谱展开也无案例。但 Neo4j 已建好 `case_record_multimodal_index` 向量索引、`RelationServiceImpl` 已有 `CaseRecord-[:RECORDED]->Fault` 关系——基础设施齐备，只差接通。
- **检索统一走向量，锚定尽力而为**：案例是自由文本经验，**主检索 = 对 `CaseRecord.multimodal_embedding` 的向量召回**（用用户故障描述/图片向量），与 Fault/Component 召回并列。因此案例**永不悬空**——`RECORDED→Fault` 与 `→Device` 边只用于**展示/过滤**，能匹配上就连、匹配不上也不阻塞，**均非强制**。

## 3. 数据模型

`CaseRecord`（Neo4j `@Node("CaseRecord")`）在现有字段基础上新增 `@Property`：

| 字段 | 取值 | 说明 |
|---|---|---|
| `status` | pending/approved/rejected | 审核状态（审核 gate） |
| `sourceType` | task/file/note_photo/voice | 来源通道 |
| `sourceTaskId` | Long? | 任务沉淀时的来源任务（防重复沉淀） |
| `sourceFileUrl` | String? | 文件/拍照/语音原件的 MinIO 地址 |
| `submittedById` | Long | 提交工人 userId |
| `reviewedById` | Long? | 审核管理员 userId |
| `reviewedAt` | LocalDateTime? | 审核时间 |
| `reviewComment` | String? | 审核意见（驳回理由等） |
| `complianceReason` | String? | 合规闸门记录（通过/拦截原因留痕） |
| `deviceId` | String? | 关联设备（来源任务带入或老师傅选；**尽力而为，非必填**） |
| `faultName` | String? | 案例对应故障文本（用于 approve 时尽力匹配/连 Fault） |

已有并复用：`title/summary/diagnosis/resolution/result/downtime/cost/experienceSummary/tags/recorder/reviewedBy/imageUrls/multimodalEmbedding`，关系 `RECORDED→Fault`。

锚定均为**尽力而为，不强制**：因为主检索走向量召回，案例不依赖任何图谱边即可被检索。设备/故障边仅用于图谱展示与过滤。

幂等：同一 `sourceTaskId` 已存在 pending/approved 的 CaseRecord 时，任务沉淀入口拦截并提示。

## 4. Python 新增端点（FixAgent）

### 4.1 `POST /ai/case/draft` — 案例起草（含 Basic Reflection）
- 入参：`{source_type, task_context?, raw_text?, images?}`。
- 处理：组装上下文 → LLM 生成结构化草稿 → **Basic Reflection 一轮**（自检「是否有遗漏/编造/不准确/与原始材料不符」→ 修订）→ 返回。
- 出参：`{title, summary, diagnosis, resolution, result, experienceSummary, tags, downtime?, cost?}`（不落库，仅草稿）。

### 4.2 `POST /ai/case/compliance` — 合规闸门
- 入参：`{text}`（案例正文拼接，语音/文件已转文字）。
- 处理：单次 LLM 结构化判定，双维度。
- 出参：`{compliant: bool, relevance: bool, legality: bool, reason: str}`。
  - `relevance`：是否属于设备检修/维修经验（过滤无关/灌水）。
  - `legality`：是否含违法/有害/敏感/人身攻击等不当内容。
  - 任一为假 → `compliant=false`，`reason` 给出可读理由。

### 4.3 （期2）文本抽取 / OCR
- 文件（txt/pdf/word）抽文本：复用 FixAgent 既有文档解析能力，产出纯文本后走 `/ai/case/draft`。
- 笔记拍照 OCR：VLM 识别图片文字，产出文本后走 `/ai/case/draft`。
- 端点形态：`POST /ai/case/extract`（入参文件/图片，出参抽取文本），或在 draft 内联，实现期再定。

### 4.4 （期3）语音
- 复用现有 ASR（`/api/asr/transcribe` 或百度）转文字 → `/ai/case/draft`。

## 5. Java 端点（weixiu，`/weixiu/case-record`）

| 方法 | 端点 | 作用 |
|---|---|---|
| POST | `/draft-from-task/{taskId}` | 组装任务上下文→调 `/ai/case/draft`→返回草稿（不落库） |
| POST | `/submit` | 跑合规闸门：过→存节点 status=pending；不过→返回 reason 拦截 |
| GET | `/pending` | 管理员待审列表 |
| POST | `/{id}/approve` | 带编辑后字段；向量化(失败阻塞)+连边(Device/Fault)→approved |
| POST | `/{id}/reject` | 带 reviewComment→rejected |
| GET | `/mine` | 工人查看自己的提交及状态 |
| POST | （期2）`/draft-from-file` | 上传文件/图片→Python 抽取/OCR→draft |

approve 细节：
1. 合规已在 submit 时通过；approve 时若管理员大改文本，可二次合规（实现期决定，默认不强制二次）。
2. **向量化（强制，检索命脉）**：`MultimodalEmbeddingUtils` 对 summary+experienceSummary(+图片) 生成 `multimodalEmbedding`，写入 CaseRecord（供 `case_record_multimodal_index` 召回）。**失败则报错、阻塞通过，允许管理员重试**——没有向量，案例就无法被 RAG 召回。
3. **连边（尽力而为，非阻塞）**：`deviceId` 有值则连 Device；`faultName` 能匹配到已有 Fault（向量/名称）则 MERGE `CaseRecord-[:RECORDED]->Fault`。**匹配不上就不连，不阻塞通过、不新建 Fault/Component 节点**（避免孤儿与重复）。
4. 置 status=approved + reviewedById/At。

### 5.5 检索与展示（沉淀的"出口"——必须接通，否则前功尽弃）

**检索（RAG，期1 必做）**：在 `GraphQueryServiceImpl.search`（path/search）中并入一路**案例向量召回**——新增 `CaseRecordService.getByEmbedding(faultDescription向量)` / `getByMultimodalEmbedding(图片向量)`，用 `case_record_multimodal_index` 召回 approved 案例，作为 `cases` 数组随诊断路径一起返回。`graph_java_tool` 把案例并入证据交给 LLM。这样 LLM 在生成检修步骤 / 检修助手答疑时能引用真实案例经验。

**展示（图谱查看器，期1 推荐）**：前端图谱展开**故障**时，除 `getFaultSolutions` 外，新增 `getFaultCases(faultId, {page,size})`（分页同现有），列出该故障 `RECORDED` 的案例；点击案例看详情。`RelationServiceImpl` 已有 `CASE_RECORD_RECORDED_FAULT` 关系，新增对应的展开查询即可。

## 6. 前端面（fix-，三期累积）

工人侧：
- 任务详情（CLOSED）「沉淀为案例」按钮 → 拉草稿 → 可编辑弹窗 → 提交（期1）。
- 工作台/知识区「上传经验」：文件上传 / 笔记拍照 / 语音口述 → 草稿弹窗 → 编辑 → 提交（期2/3）。
- 「我的提交」列表看状态（pending/approved/rejected + 驳回意见）。

管理员侧：
- 现有「知识管理/沉淀审核」页签内新增「案例审核」子页：待审列表 → 详情可编辑修正（复用 `DistillationReviewPanel` 可编辑思路）→ 通过 / 驳回。

合规拦截：submit 被拦时，前端明确展示 `reason`，允许工人修改后重提。

## 7. 三期交付与验收

### 期 1 · 主干（任务沉淀通道）+ 检索接通
范围：CaseRecord 模型扩展；`/ai/case/draft` + `/ai/case/compliance`；Java draft-from-task/submit/pending/approve/reject/mine；**案例向量召回并入 path/search（让 LLM 看得到）**；前端任务详情按钮 + 管理员案例审核子页 + 我的提交 +（推荐）故障展开列案例。
验收：CLOSED 任务沉淀为案例 → 合规闸门生效（违规/无关被拦并显示理由）→ 老师傅改→提交→管理员审核通过 → Neo4j 出现 approved CaseRecord 节点 + 向量 +（有则）Device/Fault 边；向量化失败时阻塞通过且可重试。**关键验收：审核通过后，用一句相关故障描述向 Agent 提问 / 诊断，能召回到这条案例（证明 RAG 出口接通）。**

### 期 2 · 文件 / 笔记拍照通道
范围：文件上传抽文本 + 笔记拍照 OCR → `/ai/case/draft`（Reflection）→ 走主干；前端「上传经验」文件/拍照入口。
验收：上传一份 pdf/word/txt 或拍一张笔记照片 → AI 出结构化草稿 → 老师傅改 → 主干审核入图谱。

### 期 3 · 语音口述通道
范围：复用 ASR 转文字 → `/ai/case/draft` → 走主干；前端语音入口。
验收：老师傅口述一段经验 → 转文字 → AI 整理草稿 → 主干审核入图谱。

## 8. 不做 / 边界（YAGNI）

- 不做「工人从零纯手填表单」（改为上传文件/拍照/语音，门槛低且有原件可溯源）。
- 期 2/3 才做非任务通道；期 1 仅任务沉淀。
- **不把案例里的新故障/部件/方案抽成独立节点**（避免孤儿与重复）；案例只**尽力连**已存在的 Device/Fault，连不上不阻塞。
- **不强制设备/故障**：主检索走向量召回，案例不依赖图谱边即可被 RAG/用户找到（解决了"没设备→悬空"的顾虑）。
- 案例向量召回**已纳入期 1**（不是后续增强）——否则沉淀对 LLM 不可见，等于白做。
- 案例对应的故障若图中无匹配节点，则**不建 Fault 节点**，仅以 `faultName` 文本留存在 CaseRecord 上 + 向量可检索。
