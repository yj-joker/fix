# 检修任务入口有效性校验（防垃圾守门闸）— 设计与实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: 用 superpowers:subagent-driven-development 逐任务实现，任务间主控 review。步骤用 `- [ ]` 勾选。
> **本项目无自动化测试**：验证用「`mvn -o compile` / `npm run build` 通过 + curl/走查」。

**Goal:** 在员工创建检修任务时加一道**轻量、宽松、可降级**的入口闸，挡掉乱码/无关的垃圾任务，避免触发 ~60s 的昂贵 AI 生成、污染图谱、增加管理员审核量。

**设计依据（成本不对称）：** 一次校验 = 几毫秒规则 + 最多一次便宜快模型调用(qwen-turbo, 1–2s)；挡掉的垃圾任务 = 60s 生成 + 4000 token + 图谱噪声 + 管理员审核。用 1–2s 换 60s，划算。

**定位（重要）：** 只"挡垃圾"，**不**判任务好坏（质量留给后面管理员沉淀）。**宁松勿严**——短而有效的描述（如"发动机异响""怠速不稳"）必须放过。

**依赖：** 案例子系统期1 的 `/ai/case/compliance` 已落地（本计划把它**泛化**为通用 `/ai/validate`）。故本计划在案例期1 之后执行。

参考 spec：`docs/superpowers/specs/2026-06-09-案例经验知识沉淀-design.md`

---

## 架构

```
员工创建检修任务(faultDescription)
  → [同步] Layer1 规则校验(空/过短/乱码) ── 不过 → 400+理由，不入队
  → [同步] Layer2 调 /ai/validate(purpose=task) ── 不过 → 400+理由，不入队
  → 通过 → 照常 sendGenerateMessage 触发 AI 生成
（Layer2 调用失败/超时/欠费 → fail-open 放行，校验器宕机不拖垮建任务）
```

**守门端点复用**：`/ai/validate` 由案例的 `/ai/case/compliance` 泛化而来，`purpose=case` 走相关性+合规性，`purpose=task` 走宽松相关性。任务与案例共用一套守门逻辑。

- [ ] **Step 2: case_agent.py 加任务校验函数（宽松相关性）**

```python
_TASK_VALIDATE_SYS = """判断输入是否是一个“有效的设备检修问题/故障现象”，而非乱码、空话或与检修无关的闲聊。
只输出 JSON：{"valid":bool,"reason":str}。
判定要宽松：只要像真实的设备故障/检修需求就算有效，简短描述（如“发动机异响”）也算有效；
仅当明显是乱码、测试字符、或与设备检修完全无关时才 valid=false，reason 用中文说明。"""

async def validate_task_text(text: str) -> dict:
    llm = get_llm_service()
    r = await llm.chat(
        messages=[{"role":"system","content":_TASK_VALIDATE_SYS},
                  {"role":"user","content":text[:1000]}],
        response_format={"type":"json_object"},
        model=get_settings().intent_router_model,   # 用便宜快模型 qwen-turbo
    )
    d = _safe_json(r["content"]) or {}
    return {"valid": bool(d.get("valid", True)), "reason": d.get("reason","")}
```
（`_safe_json`、`get_llm_service` 已在案例期1 引入；`get_settings().intent_router_model` 默认 qwen-turbo。）

- [ ] **Step 3: main.py 通用端点**

```python
from agents.case_agent import check_compliance, validate_task_text
from schemas.request import ValidateRequest

@app.post("/ai/validate")
async def ai_validate(req: ValidateRequest):
    if req.purpose == "case":
        c = await check_compliance(req.text)
        return {"valid": c["compliant"], "reason": c["reason"]}
    v = await validate_task_text(req.text)
    return {"valid": v["valid"], "reason": v["reason"]}
```

- [ ] **Step 2: 两层校验方法**

```java
/** 入口闸：不过抛 IllegalArgumentException(message=理由)，由全局异常→400 返回前端 */
private void validateTaskInput(String faultDescription) {
    if (!validateEnabled) return;
    // Layer1 规则（免费）
    String t = faultDescription == null ? "" : faultDescription.trim();
    if (t.length() < 4) throw new IllegalArgumentException("故障描述太短，请具体描述故障现象");
    if (t.chars().distinct().count() <= 2) throw new IllegalArgumentException("故障描述无效，请描述真实故障");
    // Layer2 快模型（可降级：调用失败则放行）
    if (!validateLlmEnabled) return;
    try {
        Map<String,Object> body = Map.of("text", t, "purpose", "task");
        String resp = webClient.post().uri("/ai/validate").bodyValue(body)
                .retrieve().bodyToMono(String.class).block();
        JsonNode node = objectMapper.readTree(resp);
        if (!node.path("valid").asBoolean(true)) {
            String reason = node.path("reason").asText("该故障描述无效或与设备检修无关");
            throw new IllegalArgumentException(reason);
        }
    } catch (IllegalArgumentException e) {
        throw e; // 校验不过，透出理由
    } catch (Exception e) {
        log.warn("[任务校验] 校验服务不可用，放行(fail-open): {}", e.getMessage());
        // fail-open：校验器宕机/欠费不拖垮建任务
    }
}
```

## Self-Review 结论
- 覆盖讨论结论：两层(规则+快模型)、宽松、可降级(fail-open + 双开关)、复用泛化 `/ai/validate`。
- 守门复用：`/ai/validate` 统一 task/case，少维护一套——与"一个守门 LLM"的架构判断一致。
- 定位克制：只挡垃圾、同步卡 create、不入队；质量审核仍归后续沉淀。
- 依赖明确：建立在案例期1 的 `case_agent` 之上，故排在其后。
