# 删除「手册自动抽取知识图谱」— 设计与实现计划

> **For agentic workers:** REQUIRED SUB-SKILL: 用 superpowers:subagent-driven-development 逐任务实现，任务间主控两阶段 review。步骤用 `- [ ]` 勾选。
> **本项目无自动化测试**：验证用「`mvn -o compile` / `npm run build` 通过 + 关键链路人工走查」。

**Goal:** 移除"手册 PDF → LLM 按章节抽取 Component/Fault/Solution → 入图谱"这条低价值、易空、与向量 RAG 重复的链路；保留手册向量 RAG、manual_device 推荐、图谱模型、任务/案例沉淀这些有价值的部分。

**为什么删（设计依据）：** 手册细节知识已由 PDF 切分进向量库覆盖（更全），手册自动抽取的图谱实体稀疏甚至空、维护成本高且与向量 RAG 重复。删后职责更清爽：**手册知识→向量 RAG；实战知识（任务沉淀+案例沉淀）→知识图谱**。图谱不会变空——`检修任务 promote-to-graph` 独立建 Device/Component/Fault/Solution 节点。

## 删 / 留 边界（已与用户确认）

### 删除
- **Python**：`services/manual_graph_extractor.py`、`services/manual_graph_ingest.py`；`mq/consumer.py` 的 `handle_kg_extract` + `knowledge.kg.extract` 队列/绑定/channel/常量；手册导入 ready 后**自发 kg.extract 的发布**（`consumer.py:271` 附近）。
- **Java**：`controller/GraphIngestController.java`（/graph/ingest、/unverified、/approve、/reject 全删）；`service/GraphIngestService.java` + `impl/GraphIngestServiceImpl.java` 整体删除；两处 `linkManualComponentsToDevice` 调用（`MaintenanceManualServiceImpl:129`、`MaintenanceTaskServiceImpl:789`）及相应注入。
- **前端**：`KnowledgeGraphExplorer.vue` 里"待审核图谱节点(unverified→approve/reject)"的 UI；`api/graph.js` 的 `listUnverified/approveSolution/rejectNode`。

### 保留（别误伤）
- 手册 PDF→向量 的整条导入与检索（RAG 命脉）。
- `manual_device` 表 + "管理员上传手册选设备"（`syncManualDevices` 的**写表**部分，推荐链路 `ManualRecommendServiceImpl` 在用）。
- Device/Component/Fault/Solution 实体、仓库、`GraphQueryService`、`RelationService`、Neo4j 向量索引。
- 检修任务 `promote-to-graph`、案例检索。

### 已定的两个微决策
- **存量数据**：Neo4j 里历史 `source=manual_auto` 节点**留着不删**（无害，甚至可用）。
- **前端审核 UI**：删（unverified 只来自手册抽取，删功能后该 UI 永远为空）。
