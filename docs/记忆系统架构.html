AI Agent 记忆系统架构

设备维修管理平台 - 三层记忆 + 向量检索 + 渐进式摘要

📉 架构总览

记忆系统由三层组成，各层职责清晰、互相配合，解决长对话中的上下文丢失问题。

📄 窗口记忆
最近 N 轮对话的原始消息。
提供即时上下文，AI直接读取。
达到阈值后触发整合。
存储: MySQL ai_message 表
📚 事实记忆
原子化、自包含的事实片段。
向量化后存入 Redis Stack。
语义检索，跨会话持久。
存储: MySQL + Redis Vector
🎯 偏好记忆
用户交互偏好（风格/格式/习惯）。
区分 explicit / inferred 来源。
explicit 直接生效，inferred 仅参考。
存储: MySQL memory_preference 表
设计原则：事实记忆是索引，原始对话才是详情来源。通过 source_seq_range 字段可回溯原始消息，实现"记忆找回"。

🔧 对话上下文组装（每轮对话前）

每次用户发消息时，Java端 AiServiceImpl.finalAiContext() 组装完整上下文发给LLM：

1. 渐进式摘要
ai_session.summary - 之前所有对话的压缩概要
2. 相关事实（向量检索）
用当前消息向量匹配最相关的5条事实
3. 用户偏好
preference_category=0 的全局偏好
4. 会话偏好
preference_category=1 的当前会话偏好
5. 窗口消息
最近 maxMemory 轮原始对话（MySQL查询）
6. 未完成事项
进行中任务、未答复问题、用户待办

向量检索流程

// Java端 searchRelevantFacts() 用户消息 "上次设备X的故障码是什么？" ↓ HTTP POST → Python /ai/memory/search_facts?query=...&top_k=5 ↓ text-embedding-v4 向量化 ↓ Redis FT.SEARCH KNN 余弦相似度 ↓ 过滤 metadata.type="fact" ↓ 返回 top-5 最相关事实 注入上下文 → "【相关历史记忆】设备X故障码E-4012，温度传感器异常..."

🔄 重试与容错机制

LLM（Qwen）偶尔返回垃圾数据（如纯数字），系统有三道防线：

防线一：Python端类型检查

# _extract_json() 中 data = json.loads(cleaned) if not isinstance(data, dict): raise ValueError(f"期望dict，实际为 {type(data).__name__}") # 防止 json.loads 解析出 float/list 等非对象类型

防线二：Python端自动重试

LLM调用
→
解析失败？重试1次
→
返回 error 状态

防线三：Java端重试

调用Python
→
失败？等3秒重试（最多2次）
→
全失败？消息保持 consolidated=0
兜底保障：getNeedIntegrationMemory() 查询所有 consolidated=0 的消息，不限轮次范围。即使本次全部失败，下次触发整合时这些消息还会被重新捞出来。消息永远不会丢失。

🎯 偏好提取策略

类型 sourceType 判定标准 存储策略 示例
显式偏好 explicit 用户直接表达偏好意愿 同类覆盖更新，立即生效 "以后回复不要写注释"
推断偏好 inferred 从行为模式推断 新增记录，仅作参考 用户反复追问细节 → 可能偏好详细回答
反例（绝对不记录为偏好）：
• 用户讨论Java代码 ≠ 偏好Java（只是当前任务涉及）
• 用户问了React问题 ≠ 偏好React前端
• 用户当前的研究主题 ≠ 兴趣偏好

⚡ 实时记忆更新（每轮对话后）

定时整合每4轮才执行一次，中间用户纠正事实或改变偏好不能及时生效。
实时更新填补这个空窗期：每轮对话后异步调用，2-3秒内完成。

定时整合（重量级）

  • 每4轮触发一次
  • 40-60秒完成
  • 提取新事实、新待办
  • 生成渐进式摘要
  • 需要 Function Calling

实时更新（轻量级）

  • 每轮对话后触发
  • 2-3秒完成
  • 只检测事实纠正 + 偏好变更
  • 不做摘要不提取新事实
  • 纯 JSON 输出，不用工具

实时更新流程

1
对话完成后异步触发
Java doOnComplete 中调用 asyncUtils.realtimeMemoryUpdate()
2
轻量级LLM判断
Python RealtimeMemoryAgent 用简短prompt让LLM判断：用户是否纠正了事实或改变了偏好？
3
如有事实纠正 → 立即更新向量库
用 wrong_content 向量匹配旧事实 → 从Redis删除旧向量 → 写入新的正确事实向量
4
Java端保存更新
事实纠正 → 保存到MySQL memory_fact
偏好变更 → explicit覆盖更新 + 清除Redis偏好缓存
下一轮对话立即使用正确信息
示例场景：
第1轮：用户说"设备X故障码是E-4012" → 提取为事实
第2轮：用户说"不对，是E-5013" → 实时检测触发 → 向量库立即更新
第3轮：AI回答相关问题时 → 向量检索命中的是正确的 E-5013
