⚡ 实时更新 (realtime_update)

  • 每轮对话后触发
  • 2-3秒完成
  • 只处理"纠正事实"和"偏好变更"
  • 不用 function calling（纯JSON输出）
  • 不阻塞主对话流
  • 是定时整合的补充，非替代

🔄 定时整合 (consolidate)

  • 每N轮（maxMemory）触发
  • 40-60秒完成
  • 完整整合：新事实、偏好、待办、摘要
  • 带 function calling（向量检索去重）
  • 有分布式锁 + 重试机制
  • 处理所有 consolidated=0 的消息
1 事实检索接口
POST /ai/memory/search_facts ⏱ ~50ms
每轮对话前，Java端用用户消息语义检索相关历史事实，注入AI对话上下文，让AI能"记住"之前的信息。
→ Java → Python 请求参数
参数类型必填说明
query String ✅ 用户当前发送的消息文本，用于语义向量匹配
top_k int ❌ 最多返回几条最相关的事实，默认5
← Python → Java 返回
{ "facts": [ { "content": "用户的设备型号是X200", "score": 0.8523, "doc_id": "fact:123:rt_1716000000", "keywords": "设备,型号,X200", "session_id": "123" } ], "query_time_ms": 45 }
返回字段类型说明
facts[]List匹配的事实列表（按相关度排序）
facts[].contentString事实文本内容
facts[].scorefloat语义相似度分数（越高越相关）

2 定时记忆整合接口
POST /ai/memory/consolidate ⏱ 40-60s
每N轮（maxMemory）对话后异步触发，做完整的记忆整合：提取新事实、发现偏好、识别待办、生成渐进式摘要。带分布式锁防并发，失败自动重试（最多2次）。

⚠️ Java端对 sourceType 的处理策略

explicit（用户明说）：查找同类偏好 → 已存在则update内容，不存在则insert新记录

inferred（推断）：直接insert新记录（不覆盖已有，可能产生重复，由后续整合去重）

3 实时记忆更新接口
POST /ai/memory/realtime_update ⏱ 2-3s
每轮对话完成后立即异步调用。轻量级LLM判断用户是否纠正了事实或改变了偏好。
如果检测到变更 → Python端立即更新向量库（事实替换），返回变更信息给Java端保存到MySQL。
→ Java → Python 请求参数
{ "session_id": "123", "user_message": "不对，故障码应该是E-5013不是E-4012", "ai_response": "AI的回复内容" }
参数类型必填说明
session_idString✅会话ID
user_messageString✅用户本轮消息（检测是否包含纠正/偏好变更）
ai_responseString❌AI本轮回复（帮助判断AI是否用了错误信息）
recent_factsString❌JSON格式的本轮注入事实列表，默认"[]"
← Python → Java 返回
{ "has_update": true, "fact_corrections": [ { "wrong_content": "故障码是E-4012", "correct_content": "故障码是E-5013", "keywords": "故障码,E-5013" } ], "preference_changes": [ { "action": "upsert", // 或 "delete" "content": "用户偏好用英文回复", "category": "交互风格", "preferenceCategory": 0, "sourceType": "explicit" } ], "superseded_fact_ids": ["fact:123:old_doc_id_1"], "latency_ms": 1850 }
返回字段类型说明
has_updateboolean是否检测到需要更新
fact_corrections[]List事实纠正列表（Python已在向量库中完成新旧替换）
  .wrong_contentString旧的错误事实描述（用于向量匹配定位）
  .correct_contentString纠正后的正确事实
  .keywordsString检索关键词
preference_changes[]List偏好变更列表（Java端负责MySQL持久化）
  .actionString"upsert"=新增或修改 / "delete"=撤销删除
  .contentStringupsert时为新偏好内容，delete时为要删除的偏好描述
  .categoryString交互风格 | 格式要求 | 工作习惯 | 关注领域 | 其他
  .preferenceCategoryint0=用户级 / 1=会话级
  .sourceTypeString永远是"explicit"（用户明确表达才会触发实时更新）
superseded_fact_ids[]List<String>被替代的旧事实向量库doc_id
（Java端用这些ID在MySQL中标记旧事实status=superseded）
latency_msint本次检测耗时（毫秒）
⚠️ Java端对 action 的处理策略

action="upsert"：查询同userId+category+preferenceCategory的已有偏好 → 存在则updateById，不存在则save新记录

action="delete"：用content模糊匹配(LIKE)找到目标偏好 → 物理删除(removeByIds)

两种操作完成后都会清除Redis偏好缓存。

4 Java端持久化处理逻辑

integrationMemory() 收到整合返回后

步骤Python返回字段Java操作目标表
1newFacts[]批量 insert（设置userId, factId=UUID, status="active"）memory_fact
2supersededIds[]按 factId IN (...) 更新 status="superseded", superseded_at=now()memory_fact
3updatedPreferences[]explicit → upsert（同类已有则update）
inferred → 直接 insert		memory_preference
4updatedUnresolved[]批量 insert（设置 consolidationSeq）memory_unresolved
5resolvedItems[]按 id IN (...) 更新 status="resolved"memory_unresolved
6briefSummary更新 session.summary 字段ai_session
7—将已处理的消息标记 consolidated=1ai_message

realtimeMemoryUpdate() 收到实时更新返回后

步骤Python返回字段Java操作目标表
1superseded_fact_ids[]按 factId IN (...) 更新 status="superseded", superseded_at=now()memory_fact
2fact_corrections[].correct_content批量 insert 新的正确事实（factId=UUID, status="active"）memory_fact
3preference_changes[] (action="delete")LIKE模糊匹配content → 物理删除 removeByIdsmemory_preference
4preference_changes[] (action="upsert")查询同类已有 → 存在则 updateById / 不存在则 savememory_preference
5—删除 Redis 偏好缓存 keyRedis
5 Redis 缓存 Key 规范
Key 模式用途TTL清除时机
Memory:Preference:{userId}:{sessionId} 偏好缓存（CachedPreferences对象，含userPrefs和sessionPrefs） 5小时 偏好变更后立即删除
Memory:Consolidation:Lock:{sessionId} 整合分布式锁（防止并发整合同一会话） 5分钟 整合完成后 finally 中删除

7 枚举值速查表
字段可选值说明
role "user" | "assistant" 消息角色
category "交互风格" | "格式要求" | "工作习惯" | "关注领域" | "其他" 偏好分类
preferenceCategory 0 = 用户级 | 1 = 会话级 0: 所有对话公用 / 1: 仅当前会话
sourceType "explicit" | "inferred" explicit: 用户直接表达 / inferred: 从行为推断
action (偏好) "upsert" | "delete" upsert: 新增或修改 / delete: 撤销删除
fact.status "active" | "superseded" active: 有效事实 / superseded: 已被替代
unresolved.status "active" | "superseded" | "resolved" active: 进行中 / superseded: 已放弃 / resolved: 已解决
unresolved.type "未答复问题" | "进行中任务" | "用户待办" 未完成事项的类型
